01 · Canvas & surface colors

The website uses a white-dominant canvas with ink-black text and three pinks as accents. Warm ivory was tried and rejected on screen — it read as nude/beige. White is the default. The header gets a barely-there pink wash so it doesn't feel sterile.

Color rule

Black is the anchor. White is the canvas. Pink-dusty is the primary accent. Light pink is the dark-theme accent. Deep pink is hover-only.

If something is reading "muddy" or "too brown," it's because #8E4F58 is being used as a fill. Move it to hover/text and replace the fill with #C49099 or #F9DDE1.

Why "+" instead of a button

The previous quick-add was a pink circle with a white plus and a soft shadow. It read as gimmicky and competed with the product. A bare "+" character at the price line is faster to recognize, doesn't compete with photography, and matches the editorial restraint of the brand.

07 · Newsletter "Be the first to know" — REBUILT MAY 2026

The above-footer email capture was rebuilt as a custom-coded theme section (sections/et-signup.liquid). It moved from a soft-pink band to an ink-black band — a deliberate "members' inner circle" moment that contrasts with the white page and leads visually into the dark footer below it. Copy leans into exclusivity and membership; no cadence promises ("one email a week" was removed).

Layout & type

  • Background: #0A0807 (ink black), full-width band, 72px top padding. A 1px ivory-at-22% divider separates it from the footer below.
  • Two-column grid: left = copy + form (flex 1.6); right = "10%" offer rail (flex 1), separated by a 1px ivory-at-12% vertical rule. Stacks on mobile (offer on top).
  • Eyebrow ("— The inner circle"): Roboto Mono 500, 10px, +0.24em, UPPER, dusty pink #C49099. Must match the offer-rail font weight — both are Roboto Mono 500, not Jost.
  • Headline ("Be the first to know."): Roboto Mono 500, 26px, +0.08em, UPPER, ivory #F5F0EC. Force the color — the theme tries to paint h2 ink-black, which is invisible on the dark band.
  • Body: Jost Light 300, 15px, ivory-soft #E4DED8, max-width 42ch. ("First dibs on new drops, 10% off your first order, and content made just for members.")
  • Offer rail: "10%" in Staatliches 120px pink #F9DDE1 (the "%" in dusty pink), clean SVG heart, "Off first order" in Roboto Mono 500 10px dusty pink.

Form, input & button

  • Email input: transparent bg, 1px border ivory-at-18%, with a pink #F9DDE1 bottom-border as the accent. Ivory text, Jost Light 14px, radius 0.
  • Autofill override (required): Chrome autofill repaints inputs blue/white. Pin it with -webkit-box-shadow: 0 0 0 1000px #0A0807 inset and -webkit-text-fill-color: #F5F0EC on :-webkit-autofill.
  • "Join us" button: pink #F9DDE1 fill, ink-black text, 4px radius, Jost Medium 11px +0.14em UPPER. Hover: transparent fill, pink text. (Note: this is the one place a 4px radius is allowed — the rest of the site is sharp 0.)
  • Error messages: friendly + per-status — empty, invalid format, rate-limited (429), and network all get their own copy in the .et-note line.

Success state & delivery

  • On submit: the form swaps to a success block — "You're on the list." (Roboto Mono 500 26px ivory) + a dashed dusty-pink "Check your inbox ♥" code chip.
  • Backend: posts client-side to the Klaviyo Client Subscriptions API (a.klaviyo.com/client/subscriptions, company id R2LGSi, revision 2024-10-15) into the Pop-up Subscribers list (SsCkHS). A 202 = success.
  • Payload gotcha: do not include a subscriptions block under profile — Klaviyo rejects it with a 400. Just type: subscription + profile email + list relationship.
Heart icon rule

Use the clean Material-style heart SVG path everywhere a heart appears (offer rail + success chip) — never an emoji ❤️, and never the old malformed path. The brand heart is one continuous filled shape, no notch artifacts.

09 · Filters & sort

Filter UI is text-first, no shadows, no swatch dots. Selected state is a pink fill, not a checkmark icon — it reads as "applied" without explanation.

Sizes filter

  • Unselected: 18×18 white square, 1px ink-black border
  • Selected: filled #C49099 (dusty pink), border #C49099, no checkmark icon
  • No shadow ever.

Color filter

  • Display as text chips, not swatch dots. Show color names (BEIGE, BLACK, BLUE), not paint splotches.
  • Font: Roboto Mono Regular 11px, +0.18em, UPPERCASE
  • Unselected: transparent bg, no border, ink-black text, 6×12 padding
  • Selected: filled #C49099, ink-black text
  • Hide the swatch image entirely (.filter-swatch-image { display: none }).

Price range slider

  • Track: 3px tall, #E4DED8 (soft ink), filled portion #0A0807
  • Thumb: 20px white circle, 2px ink border, sits on top of track (z-index 4), no shadow
  • Price range field-wrapper: single 1px ink-soft border — never nested boxes

Sort By dropdown

  • Dropdown panel: white bg, 1px ink-soft border, no box-shadow, no border-radius
  • Option labels: Jost Light 300, 13px, sentence case, ink-black
  • Selected option: color shifts to #C49099 (dusty pink)
  • Critical: the hidden radio inputs behind sort labels must stay transparent. Don't apply generic input { background:white } rules — it paints them visible and covers labels.

The Careers experience was rebuilt from scratch as native, custom-coded theme sections to kill the slow third-party hiring app. It's the brand's first long-form editorial page pattern — ivory canvas, asymmetrical/magazine layout, Staatliches as the display moment. Two pages: a values landing and a per-role detail page with an inline application form.

Editorial-page canvas exception

Long-form content pages (Careers) use warm ivory #F5F0EC as the canvas, not the white #FFFFFF used for shop/catalog surfaces. Ivory reads as "editorial / magazine" here; white stays the rule for product surfaces. This is the one sanctioned place ivory is the page background.

Landing — /pages/careers

  • Hero ("Join the team."): Staatliches clamp(64–108)px, line-height .82, ink black — set as a 7fr/5fr grid against a Jost Light 15px intro.
  • Values list (01–06): two-column grid (40px number column + content). Numbers are small Roboto Mono 500, 14px, dusty pink #C49099 — not large Staatliches/gray. Each value title is Roboto Mono 500 14px UPPER ink; body Jost Light 14px soft. Hairline rules between rows.
  • Interlude divider band: centered, 64px vertical padding, bottom border. Kicker "— Why EASTEAM" (Roboto Mono 500 10px +0.26em deep pink) over a Staatliches clamp(32–50)px pull-quote. Use this to separate sections without crowding — no image placeholders.
  • Role cards: full-width <a>, 1px ink-at-18% border, 24px padding. Title Roboto Mono 500 18px UPPER (display:block), meta Jost Light 13px soft on its own line, "Read more →" arrow Roboto Mono 10px deep pink. Hover: bg dusty-pink-at-8%. Clicking navigates to the role's own page — not a dropdown.

Role detail — /pages/{role-handle}

  • Back link: "← All roles" Roboto Mono 500 10px +0.22em deep pink → /pages/careers.

Application form (inline toggle)

Form delivery (no native Shopify contact form)

Custom forms deliver via Web3Forms (api.web3forms.com/submit, POST FormData incl. file ≤5MB), each routed to its own inbox by access key — Careers → careers@easteamny.com. Don't reuse the native Shopify contact form (it only goes to the store's default inbox and can't route per-form). Keep the honeypot botcheck field. The access key is a publishable client key — safe in theme code.

13 · Implementation (cascade & rules)

This is the technical stuff for whoever's editing CSS on the theme. The Shopify theme injects per-section CSS inline within the body, after <head>. Anything you write in head loses the cascade. The brand layer fixes this in one place.

The brand CSS layer

  • File: assets/easteam-brand.css
  • Loaded: preloaded in <head> (for FOUC), then applied via <link rel="stylesheet"> at the end of <body>
  • Loading at end of body means it cascades last, naturally winning against every section's inline CSS without needing nuclear !important everywhere.
  • Every selector uses html body prefix to guarantee specificity wins.

Specificity pattern

/* Standard EASTEAM override pattern */ html body .real-class-name, html body [id^='section-id-prefix-'] .real-class-name { property: value !important; }

The [id^='...'] attribute selector matches per-section ID-prefixed inline CSS rules — this is what the Shopify theme generates for things like slideshow blocks and marquee blocks.

Inspect, don't guess

When something isn't applying, the cause is almost always one of:

  • Wrong class name. Inspect the actual rendered HTML — the theme uses real class names like .slideshow__button, .mini-atc-btn, .section-heading-title, .product-actual-price. Don't assume .btn or .add-to-cart.
  • Inline <style> from a section. Section .liquid files inject styles like #shopify-section-XXX { ... }. Match them with [id^='shopify-section-'] .your-class.
  • CSS variable being set inline. Some sections set --node-color, --button-bg, etc. via the theme customizer's color picker. Override the variable at the same ID-prefix level with !important.
Don't break the sort dropdown

Never apply a generic input { background: white; border: 1px solid ... } rule. The theme uses hidden radio inputs overlaid on sort/filter labels — painting them visible covers the labels with empty boxes. Always scope input styling to specific forms: .product-form input, .newsletter-form input, etc.

Safe deploy workflow

How structural theme changes (new sections, page templates, rebuilds) get to the live site without risk.

  • Never edit the live theme directly. Duplicate the live theme first, build and test on the unpublished copy, then publish. (Theme-file writes to the live/MAIN theme are blocked anyway.)
  • Preview unpublished work by appending ?preview_theme_id=THEME_ID to any URL.
  • New page = three pieces. A custom page needs (1) a section .liquid file, (2) a page template templates/page.{suffix}.json that references the section, and (3) a Shopify page (handle + matching template suffix). The section must be uploaded before the template, or the template upload fails ("section does not exist").
  • Upload large files as base64. Big paste-based theme uploads occasionally corrupt (homoglyph/encoding errors); Shopify rejects the bad upload and keeps the last good version. Encode the file to base64 and verify a round-trip before upserting.

Navigation menus & the publish order

The single biggest gotcha from the Careers rebuild: a page and the link that points to it are two different things.

  • Menus are global, not per-theme. Editing a navigation menu link changes it on every theme instantly, including the live one. A menu edit is not scoped to the duplicate.
  • Publish first, then repoint. Always publish the new theme before updating a menu link to a new page. Repointing first sends live visitors to a page the live theme can't render yet.
  • A new page handle does nothing on its own. Styling /pages/careers has zero visible effect until the footer's "Career" menu item is repointed to it. The footer links live in the "ABOUT EASTEAM" menu (handle about-easteam) — they are navigation items, not hardcoded in the theme.
  • Internal links = PAGE type + resourceId. When updating a menu item to an internal page, use type: PAGE with the page's resourceId. Do not use type: HTTP with a relative /pages/x URL — Shopify mangles it to http://x (treats it as an external host). Reserve HTTP type for genuinely external URLs only.
  • menuUpdate replaces the whole menu. You must pass all items in the menu, each with its correct type/resourceId — any item you omit or downgrade gets dropped or broken.
Removing an app cleanly

When a third-party app is uninstalled (e.g. the old hiring app, UpPromote), its app-proxy URLs — like /pages/hiring/careers — go dead. Make sure no menu item or in-page link still points to them, and sweep the theme for leftover app-embed blocks in settings_data.json and orphaned snippet includes so they don't quietly slow the site.

14 · Website do's & don'ts

Do
  • Load brand CSS at end of <body> so it cascades last
  • Use html body prefix on every override
  • Inspect the rendered HTML to find real class names before writing CSS
  • Use Shopify theme customizer for section colors, copy, and layout
  • Use dusty pink #C49099 as the primary interactive accent (links, hovers, filter-selected)
  • Use Roboto Mono uppercase for all section titles and headings
  • Use Jost Light sentence case for product card titles and cart drawer items
  • Use Staatliches outlined pink for the PDP vertical marquee
  • Stack hero buttons vertically under 767px
  • Force ALL CAPS on every drawer / submenu item
  • Use a plain "+" for quick-add — no circle, no shadow, no background
  • Use ivory #F5F0EC as the canvas for long-form editorial pages (Careers) — but white stays the rule for shop surfaces
  • Override the dark newsletter input's autofill with an inset box-shadow so Chrome can't repaint it
  • Route custom forms through Web3Forms to a per-purpose inbox (Careers → careers@easteamny.com)
  • Build structural changes on a duplicate theme, then publish — never edit the live theme directly
  • Publish the new theme first, then repoint the menu link to the new page
  • Point internal menu links with PAGE type + resourceId (HTTP type is for external URLs only)
Don't
  • Don't use italics anywhere — product descriptions, body copy, captions. Brand rule.
  • Don't use #8E4F58 (deep pink) as a fill — it reads brown on screen. Hover state only.
  • Don't use ivory #F5F0EC as the page canvas — it reads as nude/peach. Use white #FFFFFF.
  • Don't add box-shadows to buttons or filter boxes.
  • Don't put any outline color on a button, in any state.
  • Don't use a checkmark icon for selected filter sizes — use a pink fill instead.
  • Don't render color filters as swatch dots — use text chips with color names.
  • Don't use uppercase Roboto Mono for product card titles — that's body content, use Jost Light.
  • Don't use Staatliches outside of the wordmark and the PDP vertical marquee.
  • Don't apply input { ... } globally — break filter radios and sort dropdown.
  • Don't add MutationObserver JS for styling — use CSS only.
  • Don't leave disabled app embed blocks in settings_data.json — remove them for speed.
  • Don't reuse the native Shopify contact form for Careers/Collab — it can't route per-form. Use Web3Forms.
  • Don't send a subscriptions block in the Klaviyo client payload — it 400s. Profile email + list relationship only.
  • Don't use an emoji heart — use the clean Material heart SVG path.
  • Don't repoint a navigation menu link before the new theme is published — menus are global and apply to the live site instantly.
  • Don't use HTTP-type menu links with a relative /pages/x URL — it mangles to http://x. Use PAGE type + resourceId.
  • Don't assume styling a new page handle changes anything — the footer link must be repointed to it separately.
Universal website rule

Black is the anchor. White is the canvas. Pink-dusty is the primary accent. No italics, no box-shadows, no outlines, no checkmarks. Consistency over cleverness. If you're tempted to add a one-off style, ask first whether the rule already exists in here.

Layout

Embedding & styling a Klaviyo form in the theme

Technique

Drop the form's own embed div where you want it: <div class="klaviyo-form-XXXXXX"></div> (the onsite script is already installed). Style it in theme CSS by targeting .klaviyo-form-XXXXXX input / textarea / button with !important — no need for Klaviyo's Custom CSS box. The form must be set to "Embed" type in Klaviyo.

CSS only ever changes appearance — submissions still flow to Klaviyo and trigger its flows (e.g. the auto-reply). Never rebuild a Klaviyo form as a native Shopify form: it routes to Shopify, not Klaviyo, and silently breaks collection + flows.

Standard button

Buttons are square (no radius), Roboto Mono, uppercase, bold, ink-black fill with light-pink #F9DDE1 text, normal height (~14px / 36px padding). Dusty-pink outline is the secondary variant. Don't go extreme on height.
